# ARCHITECTURE.md

## 1. 文档目的

本文档用于描述当前项目的整体架构、模块职责、关键数据流、部署结构与后续演进边界。

适用对象：

- 项目维护者
- AI 编码代理（如 Codex、Claude Code）
- 后续需要在本项目上继续开发、修复、重构的人

本文档的目标不是替代源码，而是提供一个稳定的“架构约束层”，避免后续改动破坏当前已经建立的实现边界。

---

## 2. 项目定位

本项目是一个面向少量熟人访问的个人展示站，部署于个人 VPS，通过 Docker 运行，并可通过 frp 对外访问。

核心定位：

- 有访问门槛的个人站点
- 不是公开内容门户
- 不是多用户平台
- 不是 CMS
- 不是复杂后台系统

核心目标：

- 访客通过全站访问密码进入首页
- 管理员通过独立 `/admin` 登录后台
- 首页展示个人入口、分组按钮、背景轮播和固定信息
- 后台可管理站点信息、按钮、背景、资源、日志
- 使用 SQLite + 本地文件存储
- 优先简单、稳定、可维护、移动端可用

---

## 3. 技术栈总览

当前项目采用单体全栈架构，核心技术如下：

- 框架：Next.js
- 语言：TypeScript
- 样式：Tailwind CSS
- 数据库：SQLite
- ORM / 数据访问：Prisma
- 资源存储：本地文件系统（uploads）
- 部署方式：Docker + docker-compose
- 运行环境：Node.js LTS
- 反向代理场景：frp / HTTPS 代理链

设计原则：

1. 优先单体应用，避免不必要的前后端拆分
2. 优先 SQLite + 本地存储，避免引入外部数据库和对象存储
3. 优先服务端控制认证与敏感逻辑
4. 优先最少依赖、最少组件层级、最少运维复杂度

---

## 4. 架构分层

项目整体可分为以下 5 层：

    用户 / 浏览器
        ↓
    Next.js 页面层（App Router）
        ↓
    API Route 层
        ↓
    领域逻辑 / lib 层
        ↓
    SQLite + 本地 uploads 存储层

### 4.1 页面层（Page / Component）

职责：

- 渲染首页、访客登录页、后台登录页、后台管理页
- 展示后台资源页、日志页等管理界面
- 发起 API 请求
- 承载交互逻辑
- 不负责最终敏感认证判断

约束：

- 页面层不能成为唯一安全边界
- 页面层不能写死密码判断逻辑
- 页面层只处理展示与交互，不直接操作数据库

---

### 4.2 API Route 层

职责：

- 接收前端请求
- 做参数校验
- 做权限校验
- 调用 `lib/` 下的领域逻辑
- 返回统一结果
- 写入访问日志或操作日志

约束：

- 所有敏感操作必须经由 API Route
- 后台相关 API 必须做管理员鉴权
- 认证相关 API 必须在服务端完成密码校验

---

### 4.3 领域逻辑层（`src/lib/*`）

职责：

- 鉴权与会话
- 密码哈希与校验
- 登录失败限制与冷却
- 站点配置读取
- 背景配置读取
- 按钮与分组数据整理
- 资源上传与删除
- 日志写入与查询
- 通用输入校验与 HTTP 响应辅助

约束：

- 领域逻辑应尽量纯粹，避免和页面耦合
- 能放在 `lib` 的逻辑，不要堆进页面组件
- 数据库访问与复杂处理优先收拢到 `lib`

---

### 4.4 数据层（Prisma + SQLite）

职责：

- 保存站点配置
- 保存密码哈希与认证设置
- 保存按钮分组与按钮
- 保存背景配置
- 保存资源记录
- 保存访问日志与操作日志

约束：

- SQLite 是当前主存储，不引入额外数据库
- schema 调整必须谨慎，优先最小必要改动
- 删除资源时必须保证数据库与本地文件联动清理

---

### 4.5 文件存储层（`storage/uploads`）

职责：

- 存放上传的图标与背景资源
- 作为 `/api/assets/:id` 的源数据

约束：

- 必须通过 Docker 挂载持久化
- 资源删除必须同步删除文件
- 禁止依赖容器内临时文件系统作为唯一存储

## 6. 核心业务模块

### 6.1 访客认证模块

职责：

- 校验访客访问密码
- 建立访客会话
- 控制首页访问门槛
- 记录登录成功/失败
- 参与失败次数限制与冷却逻辑

关键边界：

- 访客认证与管理员认证必须分离
- 访客密码必须从数据库读取
- 访客密码不得在前端明文校验

---

### 6.2 管理员认证模块

职责：

- 管理员登录 `/admin/login`
- 建立后台会话
- 保护后台页面与后台 API
- 记录管理员登录行为
- 参与失败次数限制与冷却逻辑

关键边界：

- 后台登录态与访客登录态必须隔离
- 后台接口必须校验管理员权限
- 后台密码修改后，旧会话的处理应保持一致性

---

### 6.3 站点配置模块

职责：

- 保存与读取站点标题
- 保存与读取副标题
- 保存与读取简介
- 保存与读取首页卡片样式参数
- 保存与读取背景轮播相关参数

关键边界：

- 首页所有核心文案与配置应来自数据库，而不是静态写死
- 后台修改后应能被首页读取并生效

---

### 6.4 按钮分组与按钮模块

职责：

- 管理按钮分组
- 管理按钮
- 控制显示 / 排序 / 分组归属 / 打开方式
- 为首页提供展示数据

关键边界：

- 首页展示层不应自行拼接业务规则
- 组和按钮的显示顺序应由数据层明确控制
- 删除分组时必须考虑关联按钮处理策略

---

### 6.5 背景配置模块

职责：

- 读取桌面背景组
- 读取移动端背景组
- 控制轮播间隔
- 控制随机起始
- 控制手机继承桌面背景逻辑

关键边界：

- PC 与手机背景集逻辑必须统一收口
- “手机未配置则继承桌面”是明确业务规则，不应散落在多个页面中重复实现

---

### 6.6 资源管理模块

职责：

- 处理图标 / 背景上传
- 校验格式与大小
- 对位图做基础压缩
- 对 SVG 做基础安全校验
- 提供资源读取与删除能力
- 管理资源数据库记录与本地文件联动

关键边界：

- 资源记录和文件删除必须保持一致
- `/api/assets/:id` 是当前资源访问入口
- 上传白名单与大小限制必须在服务端执行

---

### 6.7 日志模块

分为两类：

#### 访问日志

记录：

- 时间
- IP
- User-Agent
- 页面路径
- 密码是否成功
- 状态码等

#### 操作日志

记录：

- 管理员登录
- 站点配置修改
- 分组 / 按钮增删改
- 背景配置修改
- 上传 / 删除资源
- 日志清理等管理行为

关键边界：

- 日志写入逻辑应收敛在统一模块，而不是散落在每个路由中重复拼装
- 日志页是后台运维能力的一部分，不应依赖前端本地状态

---

## 7. 认证与会话设计

### 7.1 会话划分

当前存在两类会话：

1. 访客会话
2. 管理员会话

这两类会话必须逻辑隔离，不能混用。

---

### 7.2 认证流程

#### 访客流程

    访问 / 或受保护首页
        ↓
    未登录则重定向 /access
        ↓
    提交访客密码到服务端 API
        ↓
    服务端校验密码哈希
        ↓
    成功则写访客 cookie
        ↓
    允许访问首页

#### 管理员流程

    访问 /admin
        ↓
    未登录则重定向 /admin/login
        ↓
    提交管理员密码到服务端 API
        ↓
    服务端校验密码哈希
        ↓
    成功则写管理员 cookie
        ↓
    允许访问后台页与后台 API

---

### 7.3 安全边界

当前系统安全边界包括：

- 服务端密码校验
- 密码哈希存储
- 访客 / 管理员会话隔离
- 登录失败次数限制
- 登录冷却
- 后台 API 权限校验
- 访问日志与操作日志

明确不依赖以下方式作为唯一安全手段：

- 前端隐藏按钮
- 前端变量判断
- 静态页面跳转限制
- “后台地址别人不知道”这种伪安全

---

## 8. 数据模型概览

以下为逻辑实体，不要求与 Prisma 字段名完全逐字一致，但职责应稳定：

- `site_settings`：站点信息与展示参数
- `auth_settings`：密码哈希、失败计数、冷却状态等
- `button_groups`：按钮分组
- `buttons`：按钮条目
- `assets`：上传资源记录
- `background_sets`：背景配置集合
- `background_items`：背景图条目
- `access_logs`：访问日志
- `operation_logs`：操作日志

演进原则：

- schema 变更应优先小步前进
- 避免为“未来可能有用”的能力预埋过多复杂字段
- 数据模型应围绕当前项目目标，而非泛化成平台级设计

---

## 9. API 设计原则

当前 API 主要分为三类：

1. 认证类
   - 访客登录 / 退出
   - 管理员登录 / 退出

2. 后台管理类
   - 站点配置
   - 分组管理
   - 按钮管理
   - 背景管理
   - 资源管理
   - 日志管理

3. 公共能力类
   - 健康检查
   - 资源读取

设计原则：

- 后台 API 默认都视为受保护资源
- 公共 API 只开放必要能力
- 所有写操作 API 都应有基础错误处理
- 删除操作必须具备明确响应
- 重要写操作必须写操作日志

---

## 10. Docker 与部署架构

### 10.1 当前部署方式

当前推荐部署方式：

- 单个 Next.js 应用容器
- SQLite 文件落在挂载目录
- uploads 文件落在挂载目录
- docker-compose 管理
- frp 做外部入口暴露

---

### 10.2 持久化目录

必须持久化：

- `storage/data/`
- `storage/uploads/`

理由：

- SQLite 数据库存放在 `storage/data`
- 上传资源存放在 `storage/uploads`

如果这两个目录未挂载，容器重建将导致数据丢失。

---

### 10.3 健康检查

系统通过 `/api/health` 提供健康检查能力。

作用：

- 供 Docker healthcheck 使用
- 供部署后快速确认应用可用性
- 供运维排错时判断服务是否存活

---

## 11. 关键开发约束

后续继续开发时，必须遵守以下约束：

1. 不得无理由拆成前后端双项目
2. 不得无理由引入外部数据库
3. 不得无理由引入对象存储
4. 不得无理由替换认证体系
5. 不得把业务规则重新散落回页面层
6. 不得为了“更高级”而牺牲简单稳定
7. 不得在未评估兼容性的前提下重构数据模型

---

## 12. 后续可扩展方向

在不破坏当前架构的前提下，未来可以考虑扩展：

- 后台在线改密码页面
- 更细粒度日志筛选与清理
- 资源引用关系分析与删除保护
- 更完整图片处理链路
- Playwright E2E 自动化测试
- 自动备份脚本
- 更细致的后台移动端优化

这些都应建立在当前单体架构稳定的前提下进行，而不是推翻重来。

---

## 13. 结论

当前项目采用的是一种适合个人 VPS、自托管、低维护成本场景的单体全栈架构。

它的核心价值不在于“架构炫技”，而在于：

- 容易部署
- 容易维护
- 边界清晰
- 安全基础正确
- 对未来小步扩展友好

后续所有修改，都应优先保护这几个目标，而不是盲目追求复杂度。
