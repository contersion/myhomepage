# development-process.md

## 1. 文档目的

本文件用于约束本项目的开发流程，明确各阶段目标、实施顺序、每轮交付要求、验收方式，以及 AI 代理与人工协作时的推荐节奏。

本文件适用于：

- Codex Windows App
- Claude Code
- 其他在本仓库中执行开发任务的 AI 编码代理
- 人工开发者与 AI 协作开发流程

---

## 2. 总体开发原则

本项目采用“分阶段、小步推进、逐轮验收”的开发策略。

禁止采用以下方式：

- 一次性生成全部功能后再调试
- 在未分析项目结构前直接大规模写代码
- 未经过验收就继续叠加下一层功能
- 用重构替代设计
- 用大而全的架构替代简单稳定方案

推荐方式：

1. 先确认边界
2. 先搭骨架
3. 再做主链路
4. 再补后台管理
5. 再补安全、日志、测试、文档
6. 最后做视觉细化与体验优化

---

## 3. 项目目标摘要

本项目是一个个人展示站，部署在个人 VPS 上，通过 Docker + frp 提供访问。

核心目标：

- 访客需输入全站访问密码进入主站
- 管理员通过 `/admin` 登录后台
- 首页为极简二次元风，具备中间展示卡片、固定信息区、分组跳转按钮
- 背景图支持 PC / 手机分别配置、多图轮播、模糊遮罩、淡入淡出、随机起始
- 后台支持配置站点信息、按钮、背景、密码、资源、日志
- 使用 SQLite 与本地上传资源
- 具备基础安全与日志能力
- 适配手机端
- 便于长期维护

---

## 4. 推荐开发阶段

推荐分为 4 个阶段推进。

    flowchart TD
        A[阶段 0\n项目分析与方案确认] --> B[阶段 1\n项目骨架与基础模型]
        B --> C[阶段 2\n核心业务功能落地]
        C --> D[阶段 3\n安全/日志/上传/后台完善]
        D --> E[阶段 4\nDocker/文档/测试/验收/优化]

---

## 5. 阶段 0：项目分析与方案确认

### 5.1 目标

在动手编码前，先确认当前项目现状与实施策略。

### 5.2 必做事项

1. 扫描项目目录结构
2. 判断仓库类型：
   - 空仓库
   - 初始仓库
   - 已有前后端项目
3. 识别当前技术栈
4. 识别已有功能与已有约束
5. 确定推荐技术方案
6. 确定后续开发阶段划分
7. 说明本项目的实现边界

### 5.3 输出物

本阶段至少要输出：

- 当前项目结构分析
- 技术方案说明
- 目录结构方案
- 分阶段实施计划
- 首轮准备修改的文件列表

### 5.4 验收标准

只有当“项目结构”和“方案边界”已明确后，才能进入下一阶段。

---

## 6. 阶段 1：项目骨架与基础模型

### 6.1 目标

先把项目骨架、基础页面壳层、数据模型、配置机制与基本运行能力搭起来。

### 6.2 必做内容

1. 建立合理目录结构
2. 建立基础数据库模型
3. 建立数据库初始化逻辑
4. 建立 API 基础骨架
5. 建立基础前端页面壳层
6. 建立最小配置文件机制
7. 建立最小 Docker 骨架
8. 建立 README 初稿

### 6.3 至少应覆盖的数据实体

- `site_settings`
- `auth_settings`
- `button_groups`
- `buttons`
- `assets`
- `background_sets`
- `background_items`
- `access_logs`
- `operation_logs`

### 6.4 至少应建立的页面壳层

- 访客密码页
- 首页壳层
- 后台登录页
- 后台管理页壳层

### 6.5 本阶段不要求全部完成的内容

- 完整认证逻辑
- 完整后台 CRUD
- 完整上传
- 完整日志页
- 完整安全限制
- 最终 UI 精修

### 6.6 验收标准

- 项目可启动
- 数据库可初始化
- 页面壳层可访问
- 目录结构清晰
- README 有基础启动说明

---

## 7. 阶段 2：核心业务功能落地

### 7.1 目标

完成项目最关键的功能主链路，使系统具备实际可用性。

### 7.2 必做内容

#### 认证
- 访客密码验证
- 访客登录态
- 管理员登录
- 管理员登录态
- 后台接口鉴权
- 管理员退出登录

#### 站点信息
- 标题
- 副标题
- 简介

#### 首页样式配置
- 卡片透明度
- 卡片圆角
- 模糊强度

#### 分组与按钮管理
- 分组 CRUD
- 分组排序
- 分组显示控制
- 按钮 CRUD
- 按钮排序
- 按钮显示控制
- 打开方式控制

#### 图标管理
- 图标上传
- 图标 URL
- 基础校验

#### 背景管理
- PC 背景组
- 手机背景组
- 背景轮播间隔
- 淡入淡出
- 随机起始图
- 手机继承 PC 背景

### 7.3 验收标准

- 访客输入正确密码可进入首页
- 管理员可登录后台
- 后台可修改站点信息
- 后台可管理分组和按钮
- 首页能正确显示分组按钮
- 后台可管理背景图
- 首页能正确轮播背景图
- 手机端基本可用

---

## 8. 阶段 3：安全、日志、资源与后台完善

### 8.1 目标

补齐“适合实际部署”的关键能力。

### 8.2 必做内容

#### 安全
- 登录失败次数限制
- 登录冷却
- 密码哈希校验完善
- 会话边界检查
- 后台路径保护强化

#### 日志
- 访问日志写入
- 操作日志写入
- 日志分页查看
- 日志清理

#### 上传与资源处理
- 文件类型白名单
- 单文件大小限制
- 图片压缩
- SVG 基础安全校验
- 资源删除联动清理

#### 后台完善
- 错误提示优化
- 删除确认
- 资源管理体验优化
- 日志页可读性优化

### 8.3 验收标准

- 登录失败时可触发限制或冷却
- 访问日志与操作日志能写入并查看
- 上传非法文件时有清晰报错
- 删除资源后不会残留脏数据
- 后台关键流程可完整走通

---

## 9. 阶段 4：Docker、文档、测试、验收与优化

### 9.1 目标

让项目达到可部署、可交接、可复现状态。

### 9.2 必做内容

#### Docker
- Dockerfile
- docker-compose.yml
- 数据卷挂载
- 健康检查
- 环境变量整理

#### 文档
- README 完整化
- 配置说明
- 初始化说明
- 默认密码初始化方式说明
- 部署步骤说明
- 验收步骤说明

#### 测试 / 验收
至少覆盖：
- 认证流程
- 按钮 CRUD
- 背景配置读取
- 日志写入

#### 优化
- 修复明显 bug
- 修正不合理交互
- 优化关键错误提示
- 检查移动端布局

### 9.3 验收标准

- 本地可运行
- Docker 可运行
- SQLite 与 uploads 可持久化
- README 可指导他人启动项目
- 核心功能有最小验收路径

## 10. 每轮开发的标准输出格式

无论处于哪个阶段，AI 代理每一轮开发都应尽量按以下结构输出：

1. 当前项目状态总结
2. 本轮目标
3. 本轮范围
4. 准备修改/新增的文件列表
5. 实施过程
6. 修改结果总结
7. 手动验收步骤
8. 剩余问题 / 下轮建议

---

## 11. 每轮开发的禁止事项

每轮开发中，禁止以下行为：

- 未阅读项目结构直接写大段代码
- 未说明范围直接大规模改目录
- 未说明风险直接做重构
- 无理由替换已有成熟实现
- 只给代码不解释
- 修改后不提供运行或验收说明
- 引入与需求无关的复杂依赖
- 为未来“也许会用到”的场景过度设计

---

## 12. 文件变更控制建议

每轮开发前应先明确：

- 本轮会改哪些文件
- 是否会新增数据库表
- 是否会改 API 结构
- 是否会影响已有页面
- 是否需要同步更新文档
- 是否需要补测试

每轮开发后应总结：

- 实际改了哪些文件
- 是否存在未完成项
- 是否存在兼容性风险
- 下一轮应该先做什么

---

## 13. 推荐的 AI 协作节奏

推荐将开发任务切成多个连续轮次，而不是一次性全部生成。

推荐节奏如下：

### 第 1 轮
- 项目分析
- 技术方案
- 项目骨架
- 数据模型
- 页面壳层

### 第 2 轮
- 访客认证
- 后台认证
- 站点信息管理
- 分组与按钮管理
- 背景管理

### 第 3 轮
- 上传与资源管理
- 日志系统
- 安全限制
- 后台细化

### 第 4 轮
- Docker
- README
- 测试
- 最终验收
- UI 微调

---

## 14. 人工验收建议

每阶段完成后，建议人工至少验证以下内容：

### 阶段 1 后
- 项目能否启动
- 页面是否能打开
- 数据库是否初始化成功

### 阶段 2 后
- 访客密码流程是否可用
- 后台登录是否可用
- 按钮是否能管理并显示
- 背景图是否能轮播
- 手机端是否可访问

### 阶段 3 后
- 日志是否记录完整
- 登录失败限制是否生效
- 上传校验是否有效
- 删除资源是否干净

### 阶段 4 后
- Docker 能否一键启动
- 数据是否能持久化
- 文档是否足够指导部署
- 核心主链路是否可复现验收

---

## 15. 决策原则

若开发中出现实现分歧，按以下原则决策：

1. 是否满足已确认需求
2. 是否更安全
3. 是否更稳定
4. 是否更容易维护
5. 是否更适合 Docker + SQLite + 个人站
6. 是否更便于移动端适配
7. 是否更容易验收与部署

---

## 16. 结论

本项目不追求一次性做成复杂平台，而是应通过阶段化、小步迭代方式，逐步落地成一个：

- 能部署
- 能使用
- 能维护
- 能扩展
- 风格统一
- 安全边界基本正确

的个人展示站。

所有 AI 代理都应围绕这个目标推进开发，而不是偏离到无关复杂度上。
